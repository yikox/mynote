## 核心隐喻
把"三省六部"映射为个人任务流转的三个阶段：
- **中书省（草拟/收集）**：快速捕捉一切输入——灵感、待办、链接、随手记。不做筛选，全收。
- **门下省（审议/过滤）**：对中书省积压的内容做审阅、分类、打标签、判断"要不要做"。这是信息过滤的核心环节。
- **尚书省（执行/落地）**：通过审阅的事项拆解成可执行任务，进入执行列表；六部可对应不同领域（吏：习惯/学习，户：财务，礼：社交/生活，兵：健康/运动，刑：风险/问题，工：项目/创作）。

## 产品形态
- **平台**：iOS 优先，使用跨平台框架（Flutter 或 React Native）以便后期扩展 Android。
- **架构**：前后端分离。
  - 前端：iOS app（跨平台框架），负责交互、本地缓存。
  - 后端：REST/HTTP API，负责任务同步、用户数据持久化。
- **存储**：前期可用 SQLite 或 Supabase（PostgreSQL + 鉴权开箱即用），后期视规模迁移。
- **部署**：先用托管服务（Railway / Fly.io）降低运维成本。

## 关键功能
1. **快速收件箱（中书省）**：全局快捷输入，文本/链接皆可，无结构。
2. **审议队列（门下省）**：每日/定期对收件箱做"过堂"，决策保留/丢弃/延后，自动打标签分流。
3. **执行看板（尚书省 + 六部）**：按领域分栏的任务列表，支持状态（待办/进行中/完成）。
4. **三省流转动线**：一条任务可见完整流转路径（中书→门下→尚书→某部）。
5. **轻量回顾**：每周统计各部任务量与门下省驳回率，帮助个人审视精力分布。

## 差异化
不是 GTD 的复刻，而是用中国古代行政制度的"分权制衡"美学来组织信息流——
- **中书省**只管收，不评判；
- **门下省**只管审，不执行；
- **尚书省**只管做，不回头。
三权分立让"想"和"做"在时间上解耦，避免一边列清单一边焦虑。
