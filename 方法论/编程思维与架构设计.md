# 编程思维与架构设计

## 一、根

软件设计只有一个根本问题：**人的工作记忆是有限的。**

你同时能装进脑子里的东西就那么多。当你需要理解 A 才能改 B，需要记住 C 的约定才能写 D，需要在七个文件之间跳转才能追踪一个完整的逻辑流——你不是在编程，你是在和有限的工作记忆搏斗。

这就是为什么"复杂性"是万恶之源。

> 复杂性 = 任何让你难以理解和修改系统的因素

它有两个来源：

- **依赖**：一个东西改了，另一个东西也必须跟着改。不是"最好别这样"，而是"它们之间有了物理连接"，你想挣脱也挣脱不了。
- **模糊**：代码在这里，但意图不在这里。你看到了 what，看不到 why。变量名叫 `data`，方法名叫 `process`，重要的假设藏在某个开发者的脑子里，不在代码里。

这两个源头造成三种症状：

1. **变更放大**：改一个需求，动十几个地方。表面看是代码结构问题，本质是依赖像藤蔓一样爬满了系统。
2. **认知负荷**：改一行代码之前，你得先在脑子里加载一大堆背景知识。谁分配的这块内存？这个 null 从哪来的？这个 `if` 是处理哪个历史 bug 的？
3. **未知的未知**：最可怕的一种。你不知道改了会不会炸，因为依赖是不透明的，知识是分散的，后果是不可追踪的。

这三种症状只有一个受害者：**你的工作记忆。** 当它被耗尽，你就开始犯错。bug 不是粗心造成的，是认知负荷溢出造成的。

复杂性的另一个关键性质：**它是递增的。** 不是一次大爆炸，而是每次你对自己说"就加一个 `if`，没事的"——一千次之后，系统变成了没人能理解的东西。每一个小妥协都是理性的（完成任务），但累积效果是灾难性的。这就是 Gerald Weinberg 说的**二阶效应**：你为今天的方便做出一个选择，这个选择在三个月后变成了所有人的障碍。

Rich Hickey 给这种递增的纠缠起了一个精确的词：**complect（交织）**。你把两个本来独立的东西编在一起——状态和时间、值和身份、业务逻辑和 I/O——它们从此不能独立变化、不能独立理解、不能独立测试。每交织一次，系统的可能状态就翻倍。交织十次，你面对的是 2¹⁰ 种组合。人脑处理不了。

## 二、辨

所以，怎么辨别好坏？

**不要问：它用了什么设计模式？符合 SOLID 吗？**

这些问题把手段当成了目的。你要问的是四个诊断问题：

1. **要改一个需求，我需要动几个地方？** 如果一个"简单需求"碰了五个模块，依赖在蔓延。
2. **要做一个改动，我需要先理解多少东西？** 如果需要先学一套框架的内部机制、读三个基类、追踪四层继承，认知负荷太高。
3. **我能不能自信地说"这个改动是安全的"？** 如果不能——因为你不确定谁在依赖这个行为——那就是未知的未知。
4. **看到调用方式，我能不能猜出它做了什么？** 如果不能，接口没有承担起抽象的职责，模糊性太高。

这四个问题的答案指向同一个判断：**这个系统是在帮助你，还是在消耗你。**

好的系统帮你节省工作记忆。坏的系统和你的工作记忆为敌。

好设计和坏设计的分界线不是"优雅"或"技巧"，而是一个简单的经济学判断——**好设计降低未来的变化成本，坏设计增加未来的变化成本。** Sandi Metz 说得最直接：设计的目的不是为了现在看着漂亮，是为了让你以后还能继续设计。你今天写的每一行代码，都是在为明天的自己铺路或者挖坑。

## 三、道

### 增加：简单（sim-ple，一折，不缠绕）

Rich Hickey 从词源学的根上区分了两个被混为一谈的概念：

- **Simple（简单）**：sim-plex，"一折"。一件事只有一个维度、一个职责、一个理由去改变。它不是"容易"，它是"不交织"。
- **Easy（容易）**：物理上接近、心理上熟悉。你习惯的东西就是容易的，和它客观上简不简单没关系。ORM 容易上手，但它把对象世界和关系世界强行编在一起——容易，但不简单。

**增加简单性 = 把交织在一起的东西拆开。** 具体是：

- 用**值**替代可变状态。一个值不会因为你传递它而改变，它和上下文不纠缠。
- 用**纯函数**替代有副作用的函数。同样的输入永远同样的输出——它和时间不纠缠。
- 用**数据**替代类的深层继承。数据是透明的、可检查的；继承链是不透明的，需要追踪。
- 一个概念只放在一个地方。不是"尽量别重复"，而是"重复就是 bug 的温床"。

### 增加：深度（deep modules）

John Ousterhout 的核心审美标准：

> 最好的组件是**功能强大但接口简单**的组件。

Unix I/O 是典型：五个系统调用（`open`/`read`/`write`/`lseek`/`close`），背后是几十万行内核代码。接口小到能记在脑子里，但功能覆盖了几乎所有 I/O 操作。

这叫**深模块**：用简单的接口封装复杂的实现。你调它的时候脑子是轻松的，因为它帮你扛了认知负荷。

反过来，**浅模块**是那些接口比实现还复杂的东西——类名长得像一句话，构造函数有五个参数，但打开一看就是个 `new` 加个赋值。你被迫加载了一大堆"知识"（参数顺序、类型关系、命名约定），但换来的只是微不足道的功能。

**增加深度的本质是：把复杂性从接口层压到实现层，让使用者的工作记忆少装东西。**

### 增加：可组合性（compose, don't complect）

Hickey 的另一个核心区分：

- **Complect（交织）**：把两个独立的东西绑死。继承是把类型、行为、状态交织。状态是把值和时间的交织。ORM 是把对象和关系的交织。
- **Compose（组合）**：用独立的简单部件拼出复杂行为。Unix 管道是典范——`grep | sort | uniq`，每个工具只做一件事，组合后涌现出强大能力。

**增加可组合性 = 让每个部件独立工作，它们之间通过最小化的协议通信。** 部件之间知道的越少，组合的自由度越大。

### 减少：依赖

依赖 = 一个模块知道另一个模块的事。知道它的类名、方法签名、参数顺序、返回类型、内部状态。知道的越多，绑得越死。

Sandi Metz 把面向对象重新拉回 Alan Kay 的原意：**面向对象不是关于类和继承，是关于对象之间通过消息通信。** 你应该先设计消息——"这两个东西之间要说什么"——再设计对象——"谁说、谁听"。

减少依赖的方法是**让每个模块只知道它必须知道的**。不是"高内聚低耦合"这六个字，而是具体的思维动作：写一个方法的时候，问自己——"它需要知道调用者的什么？能不能再少知道一点？"

### 减少：认知负荷

认知负荷 = 做好一件事需要同时装进脑子里的信息量。

**减少认知负荷不只是"写得更清楚"，而是重新组织信息的分布。**

- 好的命名减少认知负荷：`timeoutSeconds` 比 `t` 少一个"这个 t 是什么鬼"的心理步骤。
- 好的注释减少认知负荷：不是重复代码说了什么，而是补充代码没说但需要知道的——为什么选这个算法？这个边界条件为什么是 0？
- 好的抽象减少认知负荷：把细节封在接口后面，让调用者不用读实现。
- 好的组织结构减少认知负荷：让相关的东西放在一起，让不相关的东西分开。不是因为"整洁"，是因为你不用在两个远距离的文件之间跳来跳去建立心理连接。

**认知负荷是最稀缺的资源，也是所有设计决策的最终结算货币。** 每一行代码、每一个模块、每一个接口，最终都是在问：它增加了多少认知负荷，又减少了多少？

### 减少：特殊情况

每一个 `if` 都是两个分支路径。每加一个 `if`，系统可能的状态翻倍。

Ousterhout 提出了一个极端的理念：**define errors out of existence（通过设计让错误不存在）。** 不是写好错误处理，而是重新设计，让这个错误根本不可能发生。

比如：与其四处检查 `null`，不如让返回值永远不可能为 `null`。与其处理"文件正在使用中"的异常，不如设计成操作本身就不需要独占文件。这听起来理想化，但思考方向是对的——**每消除一个特殊情况，就消除了一组认知负荷、一组测试路径、一组潜在的未知未知。**

### 减少：战术思维

Ousterhout 把程序员分成两种心智，这不是性格分类，是**时间视野的差异**：

- **战术思维**：目标 = 尽快让功能跑起来。对复杂性的态度 = "就一点，没事的"。时间视野 = 这个 sprint。
- **战略思维**：目标 = 让系统长期易于理解和修改。对复杂性的态度 = 零容忍。时间视野 = 未来数年。

战术思维不是懒，是**激励结构出了问题**。当你的绩效只看"完成了多少需求"，你就会变成战术程序员。当管理层把那个写代码极快但留下一堆坑的人当英雄（Ousterhout 叫他"战术风暴"），整个团队就会被拖垮——因为其他人要花数倍时间清理他留下的复杂性，而且看起来"做得慢"。

**减少战术思维不是靠意志力，是靠改变对"完成"的定义。** "功能跑起来了"不算完成。"功能跑起来了 + 没有增加系统复杂性 + 后来的人能理解"才算完成。这意味着每一次改动都要做额外的工作：重命名一个不清楚的变量，拆开两个不应该绑在一起的概念，给重要的设计决策写一行注释。不是大重构，是**每次离开的时候比来的时候干净一点。**

## 四、术

方向清楚了，具体怎么做？

### 接口先行，实现后行

写一个模块之前，先写好它的接口和注释。先确定"别人怎么用它"，再确定"里面怎么实现"。这是 Ousterhout 的建议，核心逻辑是：**接口是你对复杂性的承诺。** 如果你不能在接口层面说清楚这个模块做什么，那里面一定会泄漏复杂性。

这也是 Sandi Metz "消息优先"的实操版本——先设计对象之间"说什么"，再设计对象。

### 先写注释，再写代码

Ousterhout 说注释是抽象的基础。如果你不能在注释里描述清楚"这个函数做什么、边界条件是什么、为什么这样设计"，那说明你对这个抽象本身还没想清楚。注释先写，代码后写，注释是你的设计工具，不是事后补的文案。

好的注释描述代码中**不明显**的内容：为什么选这个算法而不是那个、这个 `+1` 是什么意思、这个 null 为什么在这里是合法的。它填补代码和意图之间的鸿沟。

### 做两次设计

不是真的做两遍，而是在选定一个方案之前，逼自己想一个不同的方案并比较。Ousterhout 说这有三个维度：

- 有没有更简单的方式？（减少认知负荷）
- 够不够通用？（增加深度）
- 有没有更有效率的方式？

"两次设计"的价值不在于选出更好的那个，而在于**比较过程中你会被迫思考"什么让设计变好"**——这个元认知才是真正提升设计能力的东西。

### 隔离变化，但不预判

Sandi Metz：你不确定某个职责会不会独立成一个类？没关系，先把它隔离在一个小角落里，让未来的决策不污染当前设计。不是要你预判变化（预判不了），而是要你让变化发生时只需要改一个地方。

隔离变化的本质是**让依赖可控**——你知道变化会从哪里来，提前给它留好边界。

### 组合，不继承

Hickey 和 Metz 在这个问题上完全一致，尽管来自不同传统：继承把类型、行为、状态交织在一起；组合让每个部件独立，通过最小接口通信。能用组合实现的多态就不要用继承。不是"继承不好"，是继承制造的 complection 代价通常超过它的收益。

### 始终保持一致

Ousterhout：一致性减少认知负荷。同一个概念用同一个名字；同一类问题用同一个模式；同一个约定在所有地方遵守。如果必须不一致，让不一致有充分理由，并且把这个理由写成文档。

## 五、用

日常判断，不需要背原则，就问自己：

1. 我现在写的这个东西，增加了什么？减少了什么？
   - 增加了依赖？还是减少了依赖？
   - 增加了认知负荷？还是减少了认知负荷？
   - 增加了特殊情况？还是消除了特殊情况？

2. 三个月后的我，会感谢现在的我吗？
   - 那个 `data` 变量名，三个月后我还能知道里面是什么吗？
   - 那个 `if` 分支，三个月后还有人知道它在处理什么场景吗？
   - 那个被我跳过不写的注释，三个月后还有人记得这个决策的原因吗？

3. 我现在是在战术模式还是在战略模式？
   - "赶紧搞完"——战术。停下来，喘口气，看看有没有更干净的做法。
   - "这个设计问题先记下来，下个 PR 再修"——战略。但前提是你真的会修。

4. 我是在 compose 还是在 complect？
   - 这两件事绑在一起之后，还能独立变化吗？能独立测试吗？能独立理解吗？
   - 如果不能，它们在 complecting。拆开。

---

归根到底，编程思维和架构设计不是什么玄学。它的全部内容就是：

**认识到人的工作记忆是有限的，复杂性是对它的消耗。好设计做的所有事情——封装、抽象、命名、解耦、注释、一致性——都在做同一件事：让后来的人（包括你自己）脑子里少装一点东西，就能做出安全的改动。**

增加简单、深度、可组合性。
减少依赖、认知负荷、特殊情况、战术思维。

这不是教条，是方向。每次写代码的时候朝这个方向挪一点，就是一个好程序员在做的事。
