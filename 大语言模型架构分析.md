## 2. DeepSeek V4 架构详解

### 2.1 核心架构特点

- **MLA (Multi-head Latent Attention)**：低秩 KV 压缩，显著降低 KV Cache 显存
- **CSA + HCA 混合注意力**：Compressed Sparse Attention + Heavily Compressed Attention
- **mHC (Manifold-Constrained Hyper-Connections)**：新型残差连接，解决深层数值不稳定
- **DeepSeekMoE**：256/384 路由专家 + 1 共享专家，Top-6 激活

### 2.2 V4-Pro 完整数据流

```
输入文本
   ↓
Tokenization (BPE Tokenizer, 128K vocab)
   ↓
Embedding Layer (vocab_size=128K → hidden_size=7168)
   ↓
┌────────────────────────────────────────────────────────────────┐
│  61 × Transformer Block:                                       │
│                                                                │
│  Block i (i = 0, 1, 2, ... 60):                                │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Pre-LayerNorm (RMSNorm, eps=1e-6)                      │  │
│  │    ↓                                                     │  │
│  │  MLA (Multi-head Latent Attention):                      │  │
│  │    - Q/K/V 投影到 latent_dim=512                        │  │
│  │    - RoPE (128 qk_rope_head_dim)                        │  │
│  │    - Q@K^T/√d → Softmax → @V                           │  │
│  │    - O 投影回 hidden_size=7168                          │  │
│  │    ↓                                                     │  │
│  │  残差连接: x + MLA_output                                │  │
│  │    ↓                                                     │  │
│  │  Post-LayerNorm (RMSNorm)                               │  │
│  │    ↓                                                     │  │
│  │  DeepSeekMoE FFN:                                       │  │
│  │    - Router: x → 384 expert scores                      │  │
│  │    - Top-6 Expert Selection (routed_scaling_factor=1.5) │  │
│  │    - + 1 共享专家 (所有 token 都经过)                   │  │
│  │    - 每个专家: intermediate_size=18432                  │  │
│  │    ↓                                                     │  │
│  │  残差连接: Block_input + MoE_output                     │  │
│  └──────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────┘
   ↓
Final RMSNorm → LM Head → Softmax → 输出 Token
```

### 2.3 V4-Flash 配置

| 配置项 | 值 |
|--------|-----|
| Transformer 层数 | 43 |
| 隐藏维度 | 4096 |
| Query 头数 | 64 |
| KV 头数 | 1 (GQA) |
| 路由专家数 | 256 |
| 激活专家数 | 6 + 1 shared |
| CSA 压缩率 | 4 |
| HCA 压缩率 | 128 |

### 2.4 CSA + HCA 混合注意力详解

```
标准 Transformer 注意力: O(n²) 复杂度

V4 解决方案:

CSA (Compressed Sparse Attention):
  - 轻量级索引器对所有 token 对做粗筛
  - 精选需要完整计算的 token 集合
  - 稀疏结构可训练，模型自己学习哪里需要高密度

HCA (Heavily Compressed Attention):
  - 在 MLA 基础上继续推进
  - KV 向量映射到低维潜空间
  - 推理时解压，FP4+FP8 混合精度

效果: 1M token 场景下 FLOPs 降至 V3.2 的 27%，KV Cache 降至 10%
```

## 3. GLM-5.1 架构详解

### 3.1 核心架构特点

- **GlmMoeDSA**：Gated DeltaNet 线性注意力 + 标准注意力 + 稀疏 MoE
- **Dense-then-MoE 模式**：前 3 层密集 FFN 热身，后 75 层切换到 MoE
- **DeepSeek Sparse Attention**：借鉴 DeepSeek 稀疏注意力设计
- **MTP Head**：Multi-Token Prediction 推测解码加速

### 3.2 完整数据流

```
输入文本
   ↓
Tokenization (vocab_size ≈ 200K+)
   ↓
Embedding Layer (vocab_size → hidden_size=12288)
   ↓
┌────────────────────────────────────────────────────────────────────────┐
│  78 × Transformer Block:                                               │
│                                                                        │
│  Block 0, 1, 2 (Dense 层):                                            │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │  Pre-LayerNorm → 标准 MHA (64 heads) → 残差 → Post-LN            │  │
│  │  → Dense FFN (intermediate_size=12288) → 残差                   │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│                                                                        │
│  Block 3-77 (MoE 层):                                                 │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │  Pre-LayerNorm                                                    │  │
│  │    ↓                                                              │  │
│  │  ┌────────────────────────────────────────────────────────────┐   │  │
│  │  │  GlmMoeDSA Attention (混合类型):                           │   │  │
│  │  │  - 标准 MHA / MLA / Gated DeltaNet (线性注意力)           │   │  │
│  │  └────────────────────────────────────────────────────────────┘   │  │
│  │    ↓                                                              │  │
│  │  MoE FFN:                                                         │  │
│  │    - Router → 256 expert scores                                  │  │
│  │    - Top-8 Selection (routed_scaling_factor=2.5)                │  │
│  │    - + 1 共享专家                                                │  │
│  └──────────────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────────────┘
   ↓
Final RMSNorm → MTP Head → LM Head → Softmax → 输出
```

### 3.3 Gated DeltaNet 结构

```
Gated DeltaNet = 门控 + Delta 状态更新 + 线性注意力

1. 门控机制:
   gate = sigmoid(W_gate @ x)
   → 控制哪些信息通过

2. Delta 状态更新:
   delta = W_delta @ x
   → 计算状态变化量

3. 线性注意力:
   选择性地关注历史 token 的某些维度
   而不是全连接的 O(n²) 注意力

4. 结合标准注意力:
   output = gate * delta + (1 - gate) * standard_att

优势: 长序列建模效率更高
```

### 3.4 关键配置

| 配置项 | 值 |
|--------|-----|
| Transformer 层数 | 78 |
| 隐藏维度 | 12288 |
| Attention Heads | 64 |
| KV Heads | 64 (Full attention) |
| 前 3 层 FFN | Dense (intermediate_size=12288) |
| 后 75 层 FFN | MoE (256 experts, top-8 routing) |
| 上下文长度 | 200K |

## 4. MiniMax M2.7 架构详解

### 4.1 核心架构特点

- **Lightning Attention + Softmax Attention**：混合注意力架构
- **32 个 MoE 专家**：总参数 456B，激活 45.9B
- **Full Attention + RoPE**：稳定长上下文处理
- **128K 滑动窗口**：适合长序列

### 4.2 完整数据流

```
输入文本
   ↓
Tokenization (vocab_size ≈ 200K)
   ↓
Embedding Layer (vocab_size → hidden_size=4096)
   ↓
┌────────────────────────────────────────────────────────────────────┐
│  32 × Transformer Block:                                           │
│                                                                    │
│  Block i (i = 0, 1, ... 31):                                      │
│  ┌──────────────────────────────────────────────────────────────┐  │
│  │  Pre-LayerNorm                                                │  │
│  │    ↓                                                          │  │
│  │  混合 Attention:                                               │  │
│  │    - Lightning Attention (线性 O(n)，长序列高效)              │  │
│  │    - Softmax Attention (标准全连接)                           │  │
│  │    - num_attention_heads=32, kv_heads=8 (GQA)                 │  │
│  │    - RoPE 旋转位置编码                                        │  │
│  │    ↓                                                          │  │
│  │  残差连接: x + attention_output                              │  │
│  │    ↓                                                          │  │
│  │  MoE FFN: 32 个专家，激活部分专家处理输入                      │  │
│  │    ↓                                                          │  │
│  │  残差连接: Block_input + MoE_output                           │  │
│  └──────────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────────┘
   ↓
Final RMSNorm → LM Head → Softmax → 输出
```

### 4.3 Lightning Attention 结构

```
Lightning Attention 将 Softmax Attention 分解为线性组件:

1. State Space:
   维护隐状态 s 编码历史信息
   s_t = A_t @ s_{t-1} + B_t @ x_t

2. 线性注意力计算:
   attn(x_t) = C_t @ s_t
   → 不需要 O(n²) 的 QK 矩阵

3. 与 Softmax Attention 混合:
   - Lightning: 处理局部窗口和状态压缩
   - Softmax: 处理关键全局依赖

优势: 长序列显存占用大幅降低
```

| 隐藏维度 | 4096 |
| Attention Heads | 32 |
| KV Heads | 8 (GQA) |
| Head Dim | 128 |
| 专家数 | 32 |

## 6. MLA 多头潜在注意力详解

### 6.1 标准 MHA vs MLA

```
标准 Multi-Head Attention (MHA):

输入 x (hidden_size):
   ↓
   Q, K, V 各自独立投影
   [batch, seq, num_heads, head_dim]
   ↓
   Q @ K^T → Softmax → @ V
   ↓
   O 投影回 hidden_size

问题: KV Cache 存储量大
     每个 token 存储: num_kv_heads × head_dim × 2 (K+V)

─────────────────────────────────────────────────────────────

MLA (Multi-head Latent Attention):

输入 x (hidden_size):
   ↓
┌────────────────────────────────────────────────────────────┐
│  Q 路径:                                                  │
│  x → Q_proj → [num_heads, seq_len, head_dim]              │
│  → RoPE                                                    │
│                                                          │
│  KV 路径 (低秩压缩):                                      │
│  x → K_latent_proj → [seq_len, latent_dim] → 存储         │
│  x → V_latent_proj → [seq_len, latent_dim] → 存储         │
│                                                          │
│  推理时解压:                                              │
│  K_latent → up_proj → K (实际维度)                        │
│  V_latent → up_proj → V (实际维度)                       │
└────────────────────────────────────────────────────────────┘
```

### 6.2 KV Cache 大小对比

| 模型 | 配置 | 单 token KV Cache (单层) | 1000 tokens KV Cache (61层) |
|------|------|-------------------------|---------------------------|
| 标准 MHA | 64 heads, head_dim=128, 64 KV heads | 64×128×2 = 16KB | 16KB × 1000 × 61 ≈ 976 MB |
| MLA (DeepSeek V4) | latent_dim=512 | 512×2 = 1KB | 1KB × 1000 × 61 ≈ 61 MB |
| MLA (Kimi K2.6) | latent_dim=512, kv_heads=1 | 512×2 = 1KB | 1KB × 1000 × 61 ≈ 61 MB |

**压缩比: ~16x**

## 7. MoE 混合专家架构

### 7.1 MoE 结构

```
标准 Transformer FFN:
  x → W1 → SiLU → ⊙ (W3 @ x) → W2 → output

MoE (Mixture of Experts):
  x → Router → Top-K Expert Selection → 选中专家并行计算
                                    → Weighted Sum → output

        ┌───────────────────────────────────────┐
        │           Router (门控网络)           │
        │  x → scores → softmax → top-K indices  │
        └───────────────┬───────────────────────┘
                        │
          ┌─────────────┼─────────────┬─────────────┐
          ↓             ↓             ↓             ↓
      Expert 0      Expert 1     Expert 2    ... Expert N
      (FFN)         (FFN)        (FFN)          (FFN)
          ↓             ↓             ↓             ↓
      output_0     output_1     output_2    ... output_N
          └─────────────┼─────────────┬─────────────┘
                        │
                        ↓
                  Weighted Sum → 最终输出
```

### 7.2 各模型 MoE 配置

| 模型 | 总专家数 | 共享专家 | 激活专家数 | Top-K |
|------|---------|---------|-----------|-------|
| DeepSeek V4-Pro | 384 + 1 shared | 1 | 6 + shared | Top-6 |
| DeepSeek V4-Flash | 256 + 1 shared | 1 | 6 + shared | Top-6 |
| GLM-5.1 | 256 + 1 shared | 1 | 8 + shared | Top-8 |
| MiniMax M2.7 | 32 | - | 部分激活 | - |
| Kimi K2.6 | 384 + 1 shared | 1 | 8 + shared | Top-8 |

### 7.3 MoE 的资源优势

```
总参数量大但激活参数量小:

以 DeepSeek V4-Pro 为例:
  - 总参数量: 1.6T
  - 激活参数量: 49B (约 3%)

优势:
  1. 推理成本 = 49B 模型成本，而非 1.6T
  2. 每个专家独立学习不同知识领域
  3. 存储空间大但计算量小
```

## 8. PD 分离部署架构

### 8.1 为什么需要 PD 分离

| 阶段 | 计算特点 | 资源需求 | 瓶颈 |
|------|---------|---------|------|
| **Prefill** | 密集计算，批量矩阵运算 | GPU 计算资源 | 算力 |
| **Decode** | 内存密集，KV Cache 读写 | GPU 显存带宽 | 显存容量 |

两个阶段优化方向不同，混合部署会导致资源互相制约。

### 8.2 PD 分离架构图

```
                    ┌─────────────────────────────────────┐
                    │           用户请求                   │
                    └─────────────────────────────────────┘
                                      │
                    ┌─────────────────┴─────────────────┐
                    ▼                               ▼
          ┌─────────────────┐            ┌─────────────────┐
          │  API Gateway    │            │                 │
          │  (请求路由分发)  │            │                 │
          └────────┬────────┘            └─────────────────┘
                   │                              │
                   ▼                              ▼
    ┌──────────────────────────┐   ┌──────────────────────────┐
    │     Prefill 节点群       │   │     Decode 节点群        │
    │                          │   │                          │
    │  ┌────────────────────┐  │   │  ┌────────────────────┐  │
    │  │  Batch Prefill     │  │   │  │  单请求 Decode 循环 │  │
    │  │  (批量并行计算)    │  │   │  │  (逐 Token 生成)    │  │
    │  └─────────┬──────────┘  │   │  └─────────┬──────────┘  │
    │            │              │   │            │              │
    │            ▼              │   │            │              │
    │  ┌────────────────────┐  │   │            │              │
    │  │  KV Cache 传输     │  │   │            ▼              │
    │  │  (到 Decode 节点)  │  │   │  Streaming Output        │
    │  └─────────┬──────────┘  │   └──────────────────────────┘
    └─────────────│─────────────┘
                  │
        RDMA/高速网络传输
                  │
                  ▼
    ┌──────────────────────────┐
    │    Decode 节点接收       │
    │   KV Cache + 首 Token    │
    └──────────────────────────┘
```

### 8.3 单请求完整数据流

```
用户输入: "请写一首关于春天的诗"

┌─────────────────────────────────────────────────────────────────┐
│                     Prefill 阶段                               │
│                  (在 Prefill 节点执行)                          │
├─────────────────────────────────────────────────────────────────┤
│  1. Tokenization → [100+ tokens]                              │
│  2. Embedding → [batch=1, seq=100, hidden=7168]               │
│  3. N×Transformer Block (批量计算)                             │
│     - 一次性计算完整序列的注意力                               │
│     - 批量激活 Top-6 专家                                       │
│  4. 生成 KV Cache                                              │
│  5. 传输 KV Cache + 首个 Token                                 │
└─────────────────────────────────────────────────────────────────┘
                          │
               RDMA/高速网络传输
                          │
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Decode 阶段                                 │
│                  (在 Decode 节点执行)                          │
├─────────────────────────────────────────────────────────────────┤
│  6. 接收 KV Cache → 存储到 GPU 显存                           │
│  7. 首个 Token 生成                                             │
│     ┌───────────────────────────────────────────────────────┐   │
│     │  while last_token != EOS:                            │   │
│     │      x = Embedding(last_token)                      │   │
│     │      for layer in [0, N]:                            │   │
│     │          attn = MLA(x, kv_cache[layer])  ← 读取KV  │   │
│     │          moe = MoE_FFN(x)                            │   │
│     │          kv_cache[layer].append(k, v)               │   │
│     │      end                                             │   │
│     │      next_token = softmax(lm_head(x)).argmax()     │   │
│     │      yield next_token (流式输出)                    │   │
│     │  end                                                 │   │
│     └───────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 8.4 KV Cache 传输计算

```
假设配置:
  - layers = 61
  - seq_len = 2000 (prompt长度)
  - kv_heads = 1 (GQA)
  - head_dim = 128
  - dtype = BF16 (2 bytes)

单层单 token KV:
  K: [1, 128] = 256 bytes
  V: [1, 128] = 256 bytes
  合计: 512 bytes / token / layer

完整序列:
  512 × 2000 × 61 = 62.5 MB

100 个并发请求:
  62.5 MB × 100 = 6.25 GB

传输时间 (假设 400Gbps RDMA):
  6.25 GB / 50 GB/s = 125 ms
```

### 8.5 Prefill vs Decode 节点硬件配置

```
┌────────────────────────────────────────────────────────────────┐
│                      Prefill 节点群                            │
├────────────────────────────────────────────────────────────────┤
│  目标: 高吞吐批量计算                                           │
│                                                                │
│  硬件配置:                                                     │
│  - GPU: H100 / A100 (高算力)                                   │
│  - 显存: 80GB+ (相对较小，不需要存大量 KV Cache)              │
│                                                                │
│  优化策略:                                                     │
│  - Continuous Batching (大量请求批量 Prefill)                  │
│  - Paged Attention (kv_cache 管理)                             │
│  - Tensor Parallelism (TP=8)                                   │
└────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────┐
│                       Decode 节点群                            │
├────────────────────────────────────────────────────────────────┤
│  目标: 低延迟长序列服务                                         │
│                                                                │
│  硬件配置:                                                     │
│  - GPU: H100 (大显存容量)                                      │
│  - 显存: 必须大，存储大量活跃请求的 KV Cache                   │
│                                                                │
│  优化策略:                                                     │
│  - Dynamic Batching (逐 token 级别 batching)                  │
│  - Paged Attention (显存管理)                                  │
│  - Prefix Caching (共享系统提示的 KV Cache)                    │
│  - Speculative Decoding (推测解码加速)                         │
└────────────────────────────────────────────────────────────────┘
```

### 8.6 PD 分离变体

#### 1. Prefix Caching 优化

```
              ┌──────────────┐
              │   Prefill    │
              │  (系统提示)  │
              └──────┬───────┘
                    │
        KV Cache (共享)
        ┌───────────┼───────────┐
        ▼           ▼           ▼
  ┌───────────┐ ┌───────────┐ ┌───────────┐
  │  Decode   │ │  Decode   │ │  Decode   │
  │ (请求 A)  │ │ (请求 B)  │ │ (请求 C)  │
  └───────────┘ └───────────┘ └───────────┘

  系统提示只 Prefill 一次，复用 KV Cache
```

#### 2. 投机解码配合 PD 分离

```
Prefill 节点:
  生成多个推测 Token: [t1, t2, t3, t4, t5]
  推测 Token + KV Cache → Decode 节点

Decode 节点:
  验证推测 Token (快速):
    for token in [t1, t2, t3, t4, t5]:
        if 验证通过: 接受 token
        else: 回退到正确分支，停止验证

  接受率约 80-90%，加速生成 2-3x
```
