# 上下文管理模块设计

## 设计思考

明确的上下文可以让 AI 比较准确的完成任务，减少 token 的消耗，也减少重复的时间

---

## 背景：2026 Agent 时代的上下文困境

### Agent 大爆发

2025-2026 年，AI Agent 从实验性 Demo 走向生产级部署。从 Anthropic 的 Multi-Agent Research System、Cognition 的 Devin、到 LangChain 生态的 LangGraph Agent，Agent 不再是一次性问答的 chatbot，而是在长任务中**循环推理、调用工具、累积状态**的自治系统。

### 模型上下文的核心局限

尽管模型上下文窗口持续扩张（GPT-4.1 标称 1M tokens，Llama 4 Scout 标称 10M tokens），**上下文管理仍然是 Agent 工程中最关键的瓶颈**。Cognition 甚至称其为 AI 工程师的「#1 工作」。核心局限如下：

#### 1. 有效上下文远小于标称窗口

2025 年 Paulsen 研究发现，在复杂任务上，模型的有效上下文可能比标称窗口**低 99%**。NoLiMa 基准测试（ICML 2025）表明：当移除问题与答案之间的字面关键词匹配后，11/13 个 LLM 在仅 32K tokens 时就跌到基线分数的 50% 以下。GPT-4o 从 99.3% 的近乎满分跌至 69.7%。

#### 2. Context Rot（上下文腐化）

Chroma 2025 年系统测试了 18 个前沿模型（GPT-4.1、Claude Opus 4、Gemini 2.5 Pro、Qwen3-235B 等），发现**所有模型**在上下文增长时都会出现质量退化。这不是模型能力问题，而是 Transformer 注意力机制的架构特性：

- **Lost in the Middle**（2023 Liu et al.）：模型对上下文开头和结尾的 token 关注度最高，中间的 token 「丢失」。
- **上下文填充率的分阶段退化**（2025 Veseli et al.）：当窗口填充 < 50% 时，呈 U 形曲线（偏好头尾）；当 > 50% 时，呈距离衰减（最近的 > 中间的 > 早期的）。
- **结构性文本反而更糟**：逻辑连贯的文档（如长报告）比随机排列的 haystack 更容易引发 rot——因为结构性内容产生更「可信的干扰项」。

#### 3. 注意力稀释

每多一个 token，模型在所有 token 上的注意力就被稀释一分。Sourcegraph 2026 年报告指出：Agent 在第 47 步做决策时，上下文中残留着第 1-46 步的全部痕迹，而**注意力预算有限**，如何分配这个预算决定了 Agent 的质量。

#### 4. Token 成本与延迟的线性膨胀

LLM 每次推理都要重新处理整个上下文窗口。这意味着一个 50K token 的上下文即使只有最后 500 token 发生变化，成本也等同于全量推理。Agent 从 2K token 起步，到会话后期膨胀到 25K+ token 是常态。Mem0 2026 LoCoMo 基准显示：全量上下文基线 72.9% 准确率、26K+ token、p95 延迟 17.12s；而使用记忆系统替代后达 91.6% 准确率、<7K token、p95 延迟 1.44s。

## 最简上下文设计

### 原始方案：全量历史

最简设计就是**不做任何处理**：把对话历史原封不动塞进上下文，让模型自己处理。

```
[sys prompt] + [turn1 user] + [turn1 assistant + tool_calls] + [turn1 tool_results] + [turn2 user] + ...
```

### 为什么不可持续

- Token 线性膨胀至超出窗口限制
- 成本随会话时长线性增长
- 模型有效注意力在长上下文中快速退化
- 旧信息可能干扰新任务的决策（Context Distraction）

### 最简可行方案

```
[sys prompt] + [compressed history summary] + [recent N turns (sliding window)] + [current turn]
```

- 用裁剪保证不超出硬限制
- 用摘要保留早期信息的基本轮廓
- 用滑动窗口保证最近交互的细节不被丢失

## 上下文压缩模块设计

### 压缩的定义

上下文压缩（Context Compression）是将已有的上下文内容在**保留关键信息的前提下减少 token 数量**的操作。与上下文选择（select）不同，压缩是**改写**而非丢弃；与上下文隔离（isolate）不同，压缩发生在**同一上下文内部**。

### 压缩策略

#### 1. 裁剪（Truncation）

最简单的方式：直接丢掉最早的 N 轮对话。

- **优点**：零延迟、确定性、无信息失真
- **缺点**：丢失的信息永远不可恢复；可能丢失关键决策背景
- **适用场景**：对话历史中早期内容显然不再相关的场景
- **改进方向**：不是简单砍头，而是按「信息密度」或「重要性」选择性裁切

#### 2. 摘要（Summarization）

用 LLM 将长段上下文压缩为精炼摘要。

- **增量摘要（Incremental Summarization）**：每次只对新增部分做摘要，合并到已有摘要中
- **锚定迭代摘要（Anchored Iterative Summarization）**：Factory 在 36,000 条真实工程会话上的评估表明，将新摘要合并到持久状态（而非从零重建）在准确度、完整性和连续性上均优于全量重建
- **结构化摘要（Structured Summary）**：按维度组织摘要（如任务进展、关键决策、待办项、笔记状态），而非自然语言段落

**关键权衡**：
- 摘要质量依赖摘要模型的能力（可能引入幻觉或遗漏）
- 摘要本身也消耗 token 和延迟
- 摘要后无法恢复细节（需保留「可展开」的引用）

#### 3. 折叠（Folding / Pointer）

不删除信息，而是将大块内容替换为轻量引用/指针。

- **笔记读取折叠**：同一路径多次读取只保留最近一次结果（Plexus 的 `foldNoteReads`）
- **工具结果折叠**：将长工具输出替换为摘要或状态标记
- **引用指针**：改为「见 note://path 第 N 行」，需要时再展开

**关键权衡**：
- 保留了恢复细节的能力
- 但折叠后的内容模型无法直接「看到」，可能影响推理
- 折叠策略需要感知语义（不是所有内容都可以安全折叠）

#### 4. 混合策略

实际系统通常是多层策略的组合：

```
[结构化状态快照] ← 摘要（任务进展/决策/待办）
[最近 N 轮完整对话] ← 滑动窗口
[折叠的工具历史] ← 折叠
[当前轮完整上下文] ← 不压缩
```

Plexus 的实践：`recentTurnsKept`（保留最近轮） + `foldNoteReads`（折叠重复读取） + 超预算 LLM 摘要兜底 + 压缩前状态快照注入。

### 压缩触发机制

| 机制 | 描述 | 典型配置 |
|------|------|----------|
| **阈值触发** | 当上下文 token 数达到窗口的 X% 时自动触发 | `snapshotTriggerRatio = 0.8` |
| **手动触发** | 用户通过命令主动触发 | `/compact` |
| **主动预测** | 在工具调用前预估返回大小，提前压缩腾空间 | 较复杂，少见 |

### 压缩粒度

- **消息级**：以整条 user/assistant/tool 消息为单位决定保留/压缩/丢弃
- **轮次级**：以一轮完整交互（user + assistant + tool calls + tool results）为单位
- **Token 级**：对单条消息内部的 token 级别做压缩（如 prompt compression，较激进）

推荐从**轮次级 + 消息级**混合粒度开始，token 级压缩目前不够成熟。

### 压缩时机

- **同步压缩**：在 LLM 调用前执行，阻塞当前请求，保证上下文干净
- **异步压缩**：后台执行，不阻塞，但模型可能在短暂窗口内使用未压缩的上下文
- **事后压缩**：在 LLM 调用完成后、下一轮开始前执行

同步压缩是最常见的选择，因为上下文必须在推理前准备好。

## 上下文管理的其他模块

### 上下文管理全景（Write / Select / Compress / Isolate）

LangChain (2025) 将上下文工程归纳为四类策略，压缩只是其中之一：

```
             ┌──────────────────────────────┐
             │      Context Engineering      │
             ├──────────┬──────────┬─────────┤
             │  Write   │  Select  │Compress │ Isolate │
             └──────────┴──────────┴─────────┴─────────┘
```

### 1. 上下文写入（Write / Scratchpad）

在 Agent 执行过程中，将关键信息**显式写入**上下文或外部状态。

- **Scratchpad**：Anthropic 的 `think` tool——让模型在执行过程中把中间思路写入持久化的便签，避免在后续轮次中丢失
- **In-Context State**：在系统提示/上下文顶部注入当前任务状态、关键决策、待办项
- **与 Compress 的关系**：Write 产生的内容最终也需要压缩管理；好的 Write 策略能生成更易压缩的结构化信息

### 2. 上下文选择（Select / Retrieval）

不压缩已有内容，而是**选择性地决定什么进入上下文**。

- **RAG 检索**：基于当前 query 从知识库检索最相关的内容注入
- **Reranker 重排序 + 硬截断**：检索后用 reranker 打分，只取 top-K 进入上下文（Sourcegraph 推荐做法）
- **动态工具选择**：根据任务阶段只暴露当前需要的工具定义，而非全部工具
- **与 Compress 的关系**：Select 在「入口」做减法，Compress 在「内部」做减法；两者互补

### 3. 上下文隔离（Isolate）

让不同子任务使用**独立的上下文窗口**，互不污染。

- **Sub-Agent 模式**：Anthropic 多 Agent 研究系统——每个子 Agent 拥有独立上下文，只关注窄子任务，效果优于单 Agent
- **多 Agent 路由**：Orchestrator-Workers 模式，主 Agent 分配任务，Worker 各自在隔离上下文中执行
- **上下文重置**：在新子任务开始时清空或大幅裁剪上下文
- **与 Compress 的关系**：Isolate 是比 Compress 更「激进」的策略——不是压缩，而是直接分割。但子 Agent 内部仍然需要 Compress

### 4. 上下文扩展（Extension / Augmentation）

为上下文**注入外部知识**，让其看到原本不在对话中的信息。

- **RAG（Retrieval-Augmented Generation）**：从知识库、文档中检索并注入
- **Memory 系统**：跨会话持久化记忆（Mem0 等），在会话开始时注入
- **Grounding**：注入实时数据（如搜索结果、API 数据）
- **与 Compress 的关系**：Extension 往上下文里「加东西」，Compress 往里「减东西」；好的设计需要两者协同——先扩展获取信息，再压缩腾空间，形成**呼吸式上下文节奏**：

```
[扩展] → 注入检索结果/工具输出
  ↓
[推理] → 模型使用完整上下文
  ↓
[压缩] → 将本轮结果提炼为摘要/状态
  ↓
[下一轮扩展] → 基于压缩后的上下文继续
```

### 模块关系总结

| 模块 | 核心问题 | 方向 | 发生时机 |
|------|----------|------|----------|
| **Write** | 什么信息值得留在上下文里？ | 产生 → 写入 | Agent 执行中 |
| **Select** | 什么信息应该进入上下文？ | 外部 → 上下文 | LLM 推理前 |
| **Compress** | 如何减少已有上下文的 token 数？ | 上下文内部缩减 | 触发条件满足时 |
| **Isolate** | 哪些信息应该在不同上下文中分开？ | 上下文 → 多个子上下文 | 任务分解时 |
| **Extension** | 什么外部信息需要注入？ | 外部 → 上下文（增加） | 需要额外知识时 |

### 其他值得关注的模块

- **上下文调度（Context Scheduling）**：管理多个上下文的生命周期——何时创建子上下文、何时合并结果、何时销毁
- **上下文审计（Context Audit）**：可视化当前上下文占用（按来源：系统提示 / 对话历史 / 工具结果 / 检索内容），帮助调试和优化
- **上下文回滚（Context Rollback）**：在 Agent 走错方向后回退到某个检查点，避免错误累积污染整个对话
