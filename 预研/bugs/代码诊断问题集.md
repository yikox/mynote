# 代码诊断问题集

一个辅助诊断系统，不是打分器。不问「好不好」，而是引导人去看那些值得看的地方。每一层从粗到细，每个问题后面附「为什么问这个」——那才是诊断逻辑本身。

---

## 第一层：仓库整体

### 1. 改一个需求，平均要动几个文件？

去 git log 里抽最近 20 个需求 commit，看每个需求碰了几个文件。如果多数需求只碰 1-3 个文件，依赖在可控范围。如果动不动就 8-10 个，依赖已经像藤蔓一样爬满了。

**为什么问这个：** 这是「变更放大」最直接的信号。一个需求应该是一个关注点，一个关注点应该主要落在一个模块里，一个模块应该集中在少数文件中。文件数是耦合的体温计。

### 2. 哪些文件每次改需求都会被碰到？

统计过去 6 个月每个文件的修改频次和修改原因。找出那种「不管改什么需求都会动到」的文件。

**为什么问这个：** 这些文件是依赖的汇聚点。它们知道太多东西，或者太多东西知道它们。通常是缺少抽象的产物——一个工具类装了几十个互不相关的方法，或者一个配置文件承载了所有模块的默认值。

### 3. 模块之间有循环依赖吗？

跑一下依赖分析工具，看有没有 A 依赖 B、B 也依赖 A 的情况。

**为什么问这个：** 循环依赖意味着两个模块在认知上是一个模块——你不能独立理解任何一个。它也意味着你永远不能单独测试其中一个。

### 4. 仓库里有几种命名约定？

同一个概念在不同的文件/模块里叫不同的名字？比如同一个东西在 A 文件叫 `timeout`，在 B 文件叫 `deadline`，在 C 文件叫 `maxWait`。

**为什么问这个：** 命名不一致直接增加认知负荷——读者每次遇到一个新名字都要判断「这是同一个东西还是不同的东西」。一致性是节省工作记忆的底层机制。

### 5. 一个新人从克隆仓库到做出第一个有信心的改动，需要多久？

这不是精确测量，是感觉。问最近入职的人。

**为什么问这个：** 这是认知负荷的综合指标。时间长不一定是代码多，而是知识太分散——要理解 A 才能碰 B，要问 C 才知道 D 的约定，要读三个基类才敢改一个子类。

## 第二层：模块/包

### 6. 每个模块能不能用一句话说清楚它做什么？

试着给每个模块写一句描述。如果出现了「和」、「以及」、「同时」——它在做不止一件事。如果写了三句还觉得没写全——它的边界是模糊的。

**为什么问这个：** 能用一句话描述 = 它只有一个维度、一个职责、一个理由去改变。这是「简单」（sim-ple，一折）的操作性定义。

### 7. 模块对外暴露的接口，比内部实现更简单吗？

数一下：这个模块的 public 方法有几个参数？加起来有多少个？然后打开看一下实现——接口的「大小」和实现的「大小」是什么比例？

**为什么问这个：** 这是深模块 vs 浅模块的现场判断。5 个 public 方法、总共 8 个参数，背后是 3000 行实现 = 深。15 个 public 方法、总共 40 个参数，背后是 200 行实现 = 浅。浅模块的接口本身就成了认知负荷，却没有帮你封装多少复杂性。

### 8. 要理解这个模块的一个 public 方法，你需要看几个其他文件？

随机挑一个 public 方法，追踪它的执行路径。你需要打开几个文件才能完整理解它在做什么？

**为什么问这个：** 如果追踪一个方法需要在 5 个文件之间跳转，复杂性被分散了而不是被封装了。封装不只是把东西放进不同文件——是让读者不需要跨文件就能理解一个完整的关注点。

### 9. 这个模块的测试，需要 mock 几个东西？

如果一个模块的单元测试需要 mock 3 个以上的外部依赖，它在 complecting。

**为什么问这个：** mock 的数量 = 这个模块和外部世界纠缠的程度。每多一个 mock，就是多一个它「知道」的外部东西。Sandi Metz 说依赖即知识——测试就是在这个知识付费。

### 10. 这个模块的变更，会触发多少其他模块的测试失败？

改一个模块的内部实现（不改变 public 接口），跑一下全量测试。有没有不相干的模块测试挂掉？

**为什么问这个：** 如果有，说明依赖方向错了，或者模块之间通过隐式渠道（全局状态、数据库 schema、共享内存）在通信。这是最危险的未知未知——你不知道谁在依赖你。

## 第三层：类/文件

### 11. 这个类的名字和它的内容一致吗？

只看类名，猜它有哪些方法。然后打开看。猜对了多少？

**为什么问这个：** 类名是一个承诺。如果类名叫 `OrderValidator` 但里面有发送邮件的逻辑，那就是承诺被违背了。每次违背都在增加模糊性——读者不再能信任命名系统。

### 12. 这个类的 public 方法，有几个是真正被外部调用的？

统计一下实际调用关系。哪些 public 方法只有一个调用者？哪些根本没有调用者？

**为什么问这个：** 多余的 public 方法是接口臃肿的信号。它们增加了读者的认知负荷——「我需要关心这个方法吗？」——却没有提供价值。它们也限制了修改的自由度，因为你不知道未来会不会有人用。

### 13. 这个类的依赖，是通过构造函数传入的，还是在方法里 new 的？

扫描一下代码里的 `new` 关键字出现在哪里。

**为什么问这个：** 在方法里 new 意味着依赖是硬编码的、不可替换的、不可测试的。构造函数注入意味着依赖是显式的、可替换的。Sand Metz：管理依赖的第一步，是让依赖可见。

### 14. 这个类用了继承吗？继承了几层？

如果有继承链，追踪一下：子类真的「是」父类吗？还是只是「用了」父类的一些方法？

**为什么问这个：** 继承把类型、行为、状态交织在一起。如果子类只是复用代码而不是真正的「is-a」关系，那就是在用继承做组合的事——complection。这种情况下继承是一个错误的选择。

### 15. 一个文件里有几个类？几个概念？

数一下这个文件里定义了几个 class/struct/interface。再数一下这些类涉及几个不同的业务概念。

**为什么问这个：** 一个文件应该是一个关注点。如果两个类总是在一起被修改，它们可能应该在一个文件里。如果两个类从来不同时出现，它们不应该在一个文件里。文件的边界应该反映修改的边界。

## 第四层：函数

### 16. 这个函数有几个参数？

数一下。超过 3 个，每一个参数都是在要求调用者「知道」一件事。参数越多，调用者的认知负荷越高。

**为什么问这个：** 参数 = 依赖 = 知识。函数要求调用者知道的越少，它就越简单。而且参数多通常意味着函数做了不止一件事——因为不同的事需要不同的输入。

### 17. 这个函数有几个分支路径？

数一下 `if`、`switch`、`try-catch`、`return`。圈复杂度是一个自动化的算法，但你自己数也是一样的。

**为什么问这个：** 每一个分支都是一条独立的路径，一组独立的测试用例，一个独立的认知单元。分支多的函数你读的时候必须在脑子里维护一个状态机——「现在在哪个分支？这个 else 对应的是哪个 if？」这是认知负荷的直接来源。

### 18. 这个函数有没有副作用？能看出来吗？

看函数签名和函数名。你能判断它会不会修改传入的参数、写文件、发网络请求、改全局状态吗？还是需要读实现才知道？

**为什么问这个：** 隐藏的副作用是未知未知的温床。调用者不知道它会做什么，自然不知道调用它会有什么后果。如果副作用必须存在，让它显式化——名字里带 `write`、`send`、`mutate`，而不是叫 `process`。

### 19. 这个函数返回 null 吗？调用者需要检查吗？

看返回值类型。如果是 nullable，追踪一下所有调用者——有没有漏掉 null 检查的地方？

**为什么问这个：** null 是一个特殊情况。每一个 null 检查都是一个分支，每一个漏掉的 null 检查都是一个潜在的 bug。Ousterhout 说 define errors out of existence——如果函数永远不返回 null（返回空列表、Optional、Result 类型），就消除了这个特殊情况。

### 20. 这个函数名的语义颗粒度和它的实现匹配吗？

函数名叫 `processData`？`handle`？`manage`？这些词不承载语义——它们不告诉你这个函数做什么，也不告诉你它不做什么。

**为什么问这个：** 好的函数名减少认知负荷，因为它帮你判断「我需要读这个函数的实现吗」。如果名字说了等于没说，你就被迫每次都去读实现。

## 第五层：注释和文档

### 21. 注释重复代码了吗？还是补充了代码里没有的东西？

随机抽 10 条注释。分类：「这是在说代码已经说了的东西」vs「这是在说代码没说的东西」。

**为什么问这个：** 重复代码的注释是噪音——它增加阅读时间但不增加理解。有价值的注释说 why 不说 what，说边界条件不说执行步骤，说设计决策不说法则。

### 22. 一个 public 方法的注释，能让你不读实现就正确使用它吗？

只看注释（不看实现），你能说出：这个方法的入参约束是什么？返回值在什么情况下会是空？它会抛什么异常？

**为什么问这个：** 注释是抽象的基石。如果调用者必须读实现才能正确调用，说明这个模块没有完成抽象——它把复杂性泄漏给了使用者。

### 23. 有没有过时的注释？

找那种旁边代码明显改过但注释没改的地方。注释里说的变量名已经在代码里不存在了，注释里说的逻辑和代码做的不一样。

**为什么问这个：** 过时的注释比没有注释更危险——它制造模糊性。读者会困惑：「是代码写错了还是注释忘了更新？」Ousterhout 说注释应该离代码足够近、足够简洁，这样才能和代码一起演化。

## 第六层：演化视角

### 24. 同一个文件在过去一年里被改过几次？每次改的原因是什么？

挑几个修改最频繁的文件，读它们的 commit 历史。修改原因集中在一个领域吗？还是每次改的原因都不一样？

**为什么问这个：** 原因集中 = 这个文件有一个清晰的职责，只是在持续演化。原因分散 = 这个文件是一个「杂货铺」，什么需求改东西都会碰它——依赖的汇聚点，需要拆分或重新设计。

### 25. 有没有超过一年没被修改过的复杂代码？

找那种圈复杂度高但从来没被改过的代码。它是「因为稳定所以没人动」还是「因为没人敢动所以稳定」？

**为什么问这个：** 长期不动的复杂代码可能是地雷。它承载着未知的未知——没有人知道它到底在做什么，也没有人敢改它。这种代码需要被测试覆盖或被重构，趁它还没变成考古遗迹。

### 26. 修复一个 bug 通常需要在几个地方改代码？

看看最近的 bug fix commit。每个 fix 是一个单点改动，还是在好几个文件里东改西改？

**为什么问这个：** 单点 fix = 好的封装——bug 被关在一个模块里。多点 fix = 一个概念被分散在多个地方，bug 也因此扩散。多点 fix 的模式说明系统结构没有反映问题的结构。

### 27. 团队里有没有「只有某个人敢改」的区域？

问团队每个人：「哪些代码你不敢动？」如果多个人的答案指向同一个模块，那个模块是知识的孤岛，也是风险的集中点。

**为什么问这个：** 代码的 bus factor 低 = 模糊性高——重要的设计意图没有留在代码和文档里，而是留在某个人的脑子里。那个人离职，知识就断了。

---

这些问题不期望你全部回答。把它们当作一组探针——当你接手一个新仓库、做一次技术评审、或者感觉某块代码「有问题但说不清」的时候，挑相关的问题去问。问题引导观察，观察形成判断，判断驱动改进。这就是辅助诊断系统能做的事。
